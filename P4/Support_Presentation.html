1. Présentation de la Mission

Contexte

Le lundi 14 août 2024, le chiffre d'affaires affiché dans PowerBI était de 275 186,59 €. Le lendemain (15 août), tous les magasins étaient fermés (jour férié). Le 16 août, en consultant à nouveau PowerBI, le chiffre d'affaires du 14 août était de 284 243,88 €.

Problème identifié

Une différence de 9 057,29 € (+3,29%) entre deux consultations du même indicateur pour la même date, sans nouvelle vente le 15 août.

Objectifs de l'audit

  1. ✅ Vérifier le chiffre d'affaires réel dans la base de données prototype
  2. ✅ Analyser le chiffre d'affaires par client (top 10)
  3. ✅ Analyser la part de chiffre d'affaires encaissé par employé
  4. ✅ Comprendre l'origine de la différence observée

2. Compréhension de l'Architecture de l'Entreprise

Architecture OLAP - Modèle en Étoile

L'entrepôt de données suit une architecture en étoile (Star Schema) typique des systèmes OLAP :

Composants de l'Architecture

Table de Faits Centrale : VENTE_DETAIL

  • Rôle : Stocke toutes les transactions de vente
  • Granularité : Article/Ticket (une ligne = un article dans un ticket)
  • Volume : 41 377 lignes
  • Clés étrangères : EAN, CUSTOMER_ID, Date_achat, id_employe

Tables de Dimensions

  1. DIM_PRODUIT (18 040 produits)

    • Décrit les produits vendus
    • Clé primaire : EAN (code-barres)

  2. DIM_CLIENT (2 297 clients)

    • Décrit les clients
    • Clé primaire : CUSTOMER_ID

  3. DIM_CALENDRIER (1 999 dates)

    • Dimension temporelle pour les analyses
    • Clé primaire : date (format Excel)

  4. DIM_EMPLOYE (56 employés)

    • Décrit les employés
    • Clé primaire : id_employe

Avantages de cette Architecture

✅ Performance : Jointures simples et rapides ✅ Simplicité : Structure facile à comprendre ✅ Optimisation OLAP : Idéal pour les agrégations ✅ Scalabilité : Facile à étendre avec de nouvelles dimensions

Vue d'ensemble

Le schéma relationnel suit un modèle en étoile avec :

Relations et Cardinalités

Relation Type Description

5. Résultats des Requêtes SQL

5.1. Chiffre d'affaires Total pour le 14 Août 2024

Requête SQL :

SELECT 
    ROUND(SUM(p.prix), 2) AS chiffre_affaires_total,
    COUNT(v.ID_BDD) AS nombre_ventes,
    COUNT(DISTINCT v.ID_ticket) AS nombre_tickets,
    COUNT(DISTINCT v.CUSTOMER_ID) AS nombre_clients
FROM 
    VENTE_DETAIL v
    INNER JOIN DIM_PRODUIT p ON v.EAN = p.EAN
    INNER JOIN DIM_CALENDRIER c ON v.Date_achat = c.date
WHERE 
    c.date = 45518  -- 14 août 2024

5.2. Top 10 des Clients par Chiffre d'Affaires (14 Août 2024)

Requête SQL :

SELECT 
    v.CUSTOMER_ID,
    cl.date_inscription,
    COUNT(v.ID_BDD) AS nombre_achats,
    COUNT(DISTINCT v.ID_ticket) AS nombre_tickets,
    ROUND(SUM(p.prix), 2) AS chiffre_affaires,
    ROUND(SUM(p.prix) * 100.0 / (
        SELECT SUM(p2.prix)
        FROM VENTE_DETAIL v2
        INNER JOIN DIM_PRODUIT p2 ON v2.EAN = p2.EAN
        INNER JOIN DIM_CALENDRIER c2 ON v2.Date_achat = c2.date
        WHERE c2.date = 45518
    ), 2) AS pourcentage_du_total
FROM 
    VENTE_DETAIL v
    INNER JOIN DIM_PRODUIT p ON v.EAN = p.EAN
    INNER JOIN DIM_CALENDRIER c ON v.Date_achat = c.date
    INNER JOIN DIM_CLIENT cl ON v.CUSTOMER_ID = cl.CUSTOMER_ID
WHERE 
    c.date = 45518
GROUP BY 
    v.CUSTOMER_ID, cl.date_inscription
ORDER BY 
    chiffre_affaires DESC
LIMIT 10

    • Aucune suppression de vente dans les logs

  4. Tous les INSERT sur Ventes sont datés du 14 août :

    • Les logs montrent que les ventes sont bien datées du 14 août
    • Le problème vient probablement de modifications de prix ou d'ajouts tardifs

Objectif : Valider les données au niveau de la base

Exemples de contraintes :

  • Dates dans une plage valide
  • Formats de données respectés

9.2. Système d'Audit Automatique

M3 : Triggers d'Audit pour les Modifications

Objectif : Tracer automatiquement toutes les modifications

Triggers implémentés :

  1. trg_vente_detail_insert_audit : Traçe chaque insertion dans VENTE_DETAIL
  2. trg_vente_detail_update_audit : Traçe chaque modification dans VENTE_DETAIL
  3. trg_vente_detail_delete_audit : Traçe chaque suppression dans VENTE_DETAIL
  4. trg_produit_update_prix_audit : Traçe les modifications de prix (impact sur le CA)
  5. trg_produit_insert_audit : Traçe les ajouts de produits
  6. trg_client_insert_audit : Traçe les ajouts de clients
  7. trg_client_update_audit : Traçe les modifications de clients

Fonctionnement :

  • Chaque modification est automatiquement enregistrée dans la table LOGS
  • Timestamp précis de chaque action
  • Détails des valeurs avant/après modification
  • Identification de l'utilisateur/application

9.3. Propriétés ACID

M4 : Transactions avec Rollback

Objectif : Garantir l'atomicité des opérations

Implémentation :

  • Utilisation de transactions BEGIN/COMMIT/ROLLBACK
  • Gestion des erreurs avec rollback automatique
  • Procédures stockées pour opérations critiques

Impact :

  • ✅ Cohérence garantie même en cas d'erreur
  • ✅ Pas de données partiellement mises à jour
